- Uniones Anidadas (Nested Joins)
    [Antes] Uniones sobre Alias (Inválido en Hibernate 6+): No se permite usar el alias de un join
    previo como prefijo para realizar un join anidado. Ejemplo incorrecto:

        SELECT e FROM Product2 e
        LEFT JOIN e.subfamily e_subfamily
        LEFT JOIN e_subfamily.family e_subfamily_family -- ¡Falla aquí! e_subfamily no se reconoce como ruta

    [Ahora] Uniones por Ruta Completa (Obligatorio en Hibernate 6+): Todas las uniones anidadas
    deben declarar la ruta completa y explícita partiendo de la entidad raíz (e). Ejemplo correcto:

        SELECT e FROM Product2 e
        LEFT JOIN e.subfamily e_subfamily
        LEFT JOIN e.subfamily.family e_subfamily_family -- ¡Correcto! Ruta Ruta desde el alias raíz 'e'

- Atributos no cualificados en uniones
    Si ordenas o filtras por un campo común (como name, id, description, code, etc.) y
    la consulta tiene JOINs, Hibernate lanzará la excepción: Ambiguous unqualified attribute
    reference '...' (qualify the attribute reference by an identification variable)

    Ejemplo incorrecto (falla en Hibernate 6+):

        SELECT e FROM Traveler e
        LEFT JOIN e.lastJourney e_lastJourney
        ORDER BY name -- ¡Error! 'name' existe tanto en Traveler como en Journey

    Solución: Calificar siempre con el alias de la entidad
    Todas las referencias a propiedades en las cláusulas de consulta deben ir siempre precedidas
    por el alias de su respectiva entidad (ej. e.).

    Ejemplo correcto:

        SELECT e FROM Traveler e
        LEFT JOIN e.lastJourney e_lastJourney
        ORDER BY e.name -- ¡Correcto! Se especifica 'e.name' de forma inequívoca

- Claves Compuestas (@IdClass)
  Con la actualización a Hibernate 6/7, la validación y el mapeo de claves compuestas (@IdClass)
  se han vuelto estrictamente conformes con la especificación de JPA.

  Si tienes una entidad donde uno de los campos de la clave compuesta (@Id) es una relación
  asociativa (@ManyToOne), tu clase @IdClass no debe declarar la relación como entidad ni contener
  anotaciones relacionales. Debe utilizar tipos simples.

  Sigue estos sencillos pasos para adaptar tu código:

  1. Cambia el tipo de dato de la relación por su clave primaria
  En la clase de clave (la declarada en @IdClass), cambia el tipo de campo de la relación (ej. State) por el tipo de su clave primaria correspondiente (ej. String).

  2. Elimina las anotaciones de mapeo relacional en la clase Key
  Quita todas las anotaciones como @ManyToOne, @JoinColumn, u otras de presentación/validación sobre ese campo en la clase de clave. La clase de clave compuesta debe permanecer como un POJO plano estándar.

  Ejemplo Práctico: Antes y Después
  Antes (No compatible con Hibernate 6/7)

  public class CityKey implements Serializable {

      @ManyToOne(fetch=FetchType.LAZY)
      @JoinColumn(name="STATE", referencedColumnName="ID")
      @DescriptionsList
      private State state; // ERROR: Tipo Entidad y anotaciones relacionales

      private int code;

      public State getState() { return state; }
      public void setState(State state) { this.state = state; }
      // ... hashCode y equals
  }
  Después (Adaptación estándar compatible)

  public class CityKey implements Serializable {

      private String state; // OK: Tipo String (clave primaria de State) y sin anotaciones

      private int code;

      public String getState() { return state; }
      public void setState(String state) { this.state = state; }
      // ... hashCode y equals adaptados a String
  }
  Nota: La entidad principal (ej. City.java) no requiere ningún tipo de cambio en su definición
   de campos, anotaciones @Id o @ManyToOne.

- expresión o.loquesea = true en JPQL

En versiones anteriores de Hibernate, la expresión o.paid = true resolvía el tipo izquierdo de la
comparación y aplicaba el mapeo personalizado para traducir el literal a 'S'.
A partir de Hibernate 6, los literales booleanos de HQL/JPQL se compilan directamente a
 literales booleanos del dialecto SQL (como true o 1). Esto provoca que se intente comparar
  un campo CHAR contra un booleano literal de SQL, rompiendo la compatibilidad en base de datos.

  Cómo solucionarlo
  Opción recomendada: Usar parámetros de consulta En lugar de incluir los literales true o false
  directamente en el texto de la consulta, hay que parametrizarlos. Al pasar el valor como parámetro,
   Hibernate fuerza el uso del UserType (SiNoType) para vincular y
   convertir el valor de forma segura antes de enviarlo a la base de datos:

  // INCORRECTO (Genera error en Hibernate 6+)
  Query query = XPersistence.getManager()
      .createQuery("from Invoice as o where o.paid = true");

  // CORRECTO (Funciona en todas las versiones)
  Query query = XPersistence.getManager()
      .createQuery("from Invoice as o where o.paid = :paid");
  query.setParameter("paid", true);

- Character -> String en Hibernate 7.2
  Problema: Las consultas nativas a columnas CHAR ahora devuelven Character en lugar de String.

  Solución: Reemplazar cast directo por toString():

  // Antes (Hibernate 5.6)
  String value = (String) query.getSingleResult();

  // Después (Hibernate 7.2)
  String value = query.getSingleResult().toString();
  Esto funciona porque tanto Character.toString() como String.toString() devuelven el valor esperado.

Guía de migración de Hibernate Type (5.6 a 7.2)
-----------------------------------------------
Hay otra guía para SiNoType que implementaba PrimitiveType<Boolean>, DiscriminatorType<Boolean>:
    migrate-hibernate-type-5.6-to-7.2.md
Aquí tienes los pasos simplificados para migrar tipos personalizados a la API de Hibernate 6/7:

1. Clases basadas en UserType (Mapeo a una columna)
Método sqlTypes(): Reemplázalo por public int getSqlType(). Retorna directamente el código numérico único (ej. Types.VARCHAR o Types.INTEGER) en lugar de un array int[].
Firma de nullSafeGet: Cambia la firma para recibir la posición de la columna como un entero (int position) en lugar de un array de nombres (String[] names).
Antes: resultSet.getObject(names[0])
Ahora: resultSet.getObject(position)
Descriptores: Reemplaza BooleanTypeDescriptor por BooleanJavaType y CharTypeDescriptor por CharJdbcType.
2. Clases basadas en CompositeUserType (Mapeo a múltiples columnas)
Eliminación de métodos de metadatos: Elimina por completo los métodos obsoletos getPropertyNames() y getPropertyTypes().
Uso de @Embeddable: Define una clase estática anidada o interna anotada con @jakarta.persistence.Embeddable que represente el grupo de columnas. Ordénalas alfabéticamente por nombre de propiedad.
Implementación de embeddable(): Sobrescribe embeddable() para que devuelva la clase @Embeddable definida.
Métodos getPropertyValue e instantiate:
Hibernate asigna índices posicionales a los atributos del @Embeddable en orden alfabético.
Implementa instantiate(ValueAccess values) para leer de forma segura dichos índices con values.getValue(index, Tipo.class).
Eliminación de lectura/escritura JDBC manual: Ya no necesitas escribir lógica en nullSafeGet o nullSafeSet, ya que Hibernate mapea los campos del @Embeddable de forma nativa.
3. Interfaces eliminadas
PrimitiveType: Quítala de la declaración de tus clases; ya no existe en el sistema de tipos de Hibernate 6+.
DiscriminatorType: Quítala; sus funcionalidades han sido integradas en mecanismos internos del motor de persistencia

Paso 2: Resolver ambigüedad de anotación View
Los archivos que usan import org.hibernate.annotations.*; causan ambigüedad
entre org.openxava.annotations.View y org.hibernate.annotations.View.

java
// ANTES
import org.hibernate.annotations.*;

// DESPUÉS
import org.hibernate.annotations.Columns;
import org.hibernate.annotations.GenericGenerator;
import org.hibernate.annotations.Parameter;
import org.hibernate.annotations.Type;
