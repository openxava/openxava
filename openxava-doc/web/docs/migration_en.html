How to update your project to the latest OpenXava?

Edit the pom.xml file in your project and change the value of the openxava.version property to point to the latest OpenXava version, thus:

<properties>
    <openxava.version>8.0</openxava.version>
    ...
</properties>

Rebuild your project with: mvn clean package

If your current version is prior to 8.0 you should follow the instructions in the section Migration from OpenXava 7.7.3 to OpenXava 8.0 first.

Finally, revise the instructions below to adapt your code.

Migration from OpenXava 7.7.3 to OpenXava 8.0

Java 17 as minimum version

Now Java 17 is the minimum supported version. If your project still uses Java 8 or 11 you should update it to compile with Java 17, 21 or 25. To do this modify the pom.xml:

<properties>
    <maven.compiler.source>17</maven.compiler.source>
    <maven.compiler.target>17</maven.compiler.target>
    ...
</properties>

Make sure you have a JDK 17 (or higher) installed on your system and configured in your IDE.

You can consult the official Java documentation for migration to Java 17 and if you use Java 8 the migration documentation from Java 8.

Jakarta Persistence 3.2 with Hibernate 7.2

OpenXava 8.0 uses Jakarta Persistence 3.2 and Hibernate 7.2. This implies significant changes compared to previous versions. Below we detail the most common issues you may encounter when migrating your application.

You can consult the official Hibernate migration guide for more information.

Change from javax.persistence to jakarta.persistence

The main change is replacing the javax namespace with jakarta in all JPA annotations. This affects your entities, key classes, and any code using persistence annotations.

// BEFORE
import javax.persistence.Entity;
import javax.persistence.Id;
import javax.persistence.ManyToOne;

// AFTER
import jakarta.persistence.Entity;
import jakarta.persistence.Id;
import jakarta.persistence.ManyToOne;

Also, if you use import org.hibernate.annotations.*; in your files, this can cause ambiguity between org.openxava.annotations.View and org.hibernate.annotations.View. The solution is to use specific imports:

// BEFORE
import org.hibernate.annotations.*;

// AFTER
import org.hibernate.annotations.Columns;
import org.hibernate.annotations.GenericGenerator;
import org.hibernate.annotations.Parameter;
import org.hibernate.annotations.Type;

persistence.xml

The persistence.xml file requires updating the namespace to version 3.1 of Jakarta Persistence:

<persistence version="3.1" 
    xmlns="https://jakarta.ee/xml/ns/persistence"
    xmlns:xsi="https://www.w3.org/2001/XMLSchema-instance"
    xsi:schemaLocation="https://jakarta.ee/xml/ns/persistence
    https://jakarta.ee/xml/ns/persistence/persistence_3_1.xsd">

NonBatchingBatcherFactory removed

The hibernate.jdbc.factory_class property with value org.hibernate.jdbc.NonBatchingBatcherFactory is no longer valid in Hibernate 7. Remove this property from your persistence.xml.

@Column(scale/precision) and @Digits on float/double types

Hibernate 7 no longer allows using @Column(scale=..., precision=...) or @Digits(integer=..., fraction=...) on float or double types:

// ERROR in Hibernate 7
@Column(scale=7, precision=7)
private double axisY;

Solution: Use @DecimalMax from Jakarta Validator instead:

// Valid solution
@DecimalMax("0.9999999")
private double axisY;

Nested Joins

In Hibernate 6+ nested joins must use the full path from the root entity. You cannot use the alias of a previous join as a prefix for a nested join.

// INCORRECT (Fails in Hibernate 6+)
SELECT e FROM Product2 e
LEFT JOIN e.subfamily e_subfamily
LEFT JOIN e_subfamily.family e_subfamily_family -- Error!

// CORRECT (Full path from root entity)
SELECT e FROM Product2 e
LEFT JOIN e.subfamily e_subfamily
LEFT JOIN e.subfamily.family e_subfamily_family -- Correct!

Unqualified attributes in JPQL queries

If you sort or filter by a common field (like name, id, description, code, etc.) and the query has JOINs, Hibernate will throw the Ambiguous unqualified attribute reference exception. You must always qualify property references with the entity alias:

// INCORRECT (Fails in Hibernate 6+)
SELECT e FROM Traveler e
LEFT JOIN e.lastJourney e_lastJourney
ORDER BY name -- Error! 'name' exists in both Traveler and Journey

// CORRECT
SELECT e FROM Traveler e
LEFT JOIN e.lastJourney e_lastJourney
ORDER BY e.name -- Correct!

Composite keys (@IdClass) with relationships

With Hibernate 6/7, @IdClass classes containing fields that are @ManyToOne relationships must be adapted:

  1. Change the field type from the relationship to its primary key (e.g. String instead of State)
  2. Remove the relationship mapping annotations (@ManyToOne, @JoinColumn, etc.) from the key class
// BEFORE (Not compatible with Hibernate 6/7)
public class CityKey implements Serializable {
    @ManyToOne(fetch=FetchType.LAZY)
    @JoinColumn(name="STATE", referencedColumnName="ID")
    private State state;
    private int code;
}

// AFTER (Compatible)
public class CityKey implements Serializable {
    private String state; // Primary key type of State
    private int code;
}

The main entity does not require changes to its @Id or @ManyToOne annotations.

Boolean expressions in JPQL

In Hibernate 6+ boolean literals are compiled directly to SQL dialect boolean literals, which can break compatibility with databases that use CHAR types for booleans (like OpenXava with SiNoType).

// INCORRECT (Generates error in Hibernate 6+)
Query query = XPersistence.getManager()
    .createQuery("from Invoice as o where o.paid = true");

// CORRECT (Use parameters)
Query query = XPersistence.getManager()
    .createQuery("from Invoice as o where o.paid = :paid");
query.setParameter("paid", true);

Custom types (UserType/CompositeUserType)

Custom types require several changes in Hibernate 7:

  • PrimitiveType/DiscriminatorType: Remove these interfaces from your class declarations
  • @TypeDef: Removed, use @Convert or @CompositeType
  • @Type(type="org.hibernate.type.YesNoType"): Change to @Convert(converter = org.openxava.jpa.converters.YesNoConverter.class)
  • @Type(type="org.openxava.types.SiNoType"): Change to @Type(org.openxava.types.SiNoType.class)

Custom types (UserType and CompositeUserType) practically need to be rewritten for the new Hibernate 7 API. The changes are deep and include method removal, signature changes, and new use of @Embeddable. Consult the Hibernate documentation for complete details.

For CompositeUserType with multiple columns, change @Columns to @AttributeOverrides:

// BEFORE
@CompositeType(org.openxava.types.Date3Type.class)
@Columns(columns = {
    @Column(name="YEARDELIVERY"),
    @Column(name="MONTHDELIVERY"),
    @Column(name="DAYDELIVERY")
})

// AFTER
@CompositeType(org.openxava.types.Date3Type.class)
@AttributeOverrides({
    @AttributeOverride(name="year", column=@Column(name="YEARDELIVERY")),
    @AttributeOverride(name="month", column=@Column(name="MONTHDELIVERY")),
    @AttributeOverride(name="day", column=@Column(name="DAYDELIVERY"))
})

Character to String in native queries

Native queries to CHAR columns now return Character instead of String. Use toString() to convert:

// Before (Hibernate 5.6)
String value = (String) query.getSingleResult();

// After (Hibernate 7.2)
String value = query.getSingleResult().toString();

@OrderBy with nested paths on associations

Hibernate 7.2 strictly enforces the JPA specification for the @OrderBy annotation. Using a nested path (such as association.property) in @OrderBy will throw a NullPointerException on startup during the SessionFactory building.

According to the JPA specification, @OrderBy must refer directly to basic attributes of the target entity. If you need to sort a collection based on an attribute of an associated entity, you must expose that column in the target entity as a read-only field and sort by it.

// INCORRECT (Fails with NullPointerException in Hibernate 7.2)
@OneToMany(mappedBy="role")
@OrderBy("module.name asc")
private Collection<ModuleRights> modulesRights;

// CORRECT
// 1. Map the join column as a read-only basic property in the target entity (ModuleRights):
@Column(name="modules_name", insertable=false, updatable=false)
private String moduleName;

// 2. Sort by the basic property in the parent entity:
@OneToMany(mappedBy="role")
@OrderBy("moduleName asc")
private Collection<ModuleRights> modulesRights;

Jakarta Validator 3.1 with Hibernate Validator 9.1

OpenXava 8.0 uses Jakarta Validator 3.1 and Hibernate Validator 9.1. This implies changes in validation annotations.

You can consult the official Hibernate Validator migration guide for more information.

Change from javax.validation to jakarta.validation

Change the validation imports:

// BEFORE
import javax.validation.constraints.*;
import javax.validation.Payload;

// AFTER
import jakarta.validation.constraints.*;
import jakarta.validation.Payload;

Jakarta Servlet 6.1 with Tomcat 11

OpenXava 8.0 has migrated its entire web layer to the modern Jakarta Servlet 6.1 specification running on Tomcat 11. This requires adjusting any custom servlet components, JSP views, and web configurations in your application.

Change from javax.servlet to jakarta.servlet

The standard Java Servlet classes have changed their package namespace. If your application defines custom filters, servlets, listeners, or other servlet-related classes, you must rename the imports:

// BEFORE
import javax.servlet.*;
import javax.servlet.http.*;
import javax.servlet.jsp.*;

// AFTER
import jakarta.servlet.*;
import jakarta.servlet.http.*;
import jakarta.servlet.jsp.*;

web.xml

Your web.xml file must be updated to reference the Servlet 6.1 namespace. Update the root tag in src/main/webapp/WEB-INF/web.xml as follows:

<web-app xmlns="https://jakarta.ee/xml/ns/jakartaee"
         xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
         xsi:schemaLocation="https://jakarta.ee/xml/ns/jakartaee 
         https://jakarta.ee/xml/ns/jakartaee/web-app_6_1.xsd"
         version="6.1">

JSTL and JSP Taglibs

Standard JavaServer Pages Standard Tag Library (JSTL) URIs have changed from their legacy sun.com locations to the new jakarta.tags namespace. If you have custom JSP files, you must update the taglib directives:

<%-- BEFORE --%>
<%@ taglib uri="http://java.sun.com/jsp/jstl/core" prefix="c" %>
<%@ taglib uri="http://java.sun.com/jsp/jstl/fmt" prefix="fmt" %>
<%@ taglib uri="http://java.sun.com/jsp/jstl/functions" prefix="fn" %>

<%-- AFTER --%>
<%@ taglib uri="jakarta.tags.core" prefix="c" %>
<%@ taglib uri="jakarta.tags.fmt" prefix="fmt" %>
<%@ taglib uri="jakarta.tags.functions" prefix="fn" %>

Commons FileUpload 2

OpenXava 8.0 has upgraded the file upload library to Apache Commons FileUpload 2. This version includes important changes in its package names and API design, particularly around the FileItem interface which is exposed in several OpenXava public APIs (such as WithFileItem, IProcessLoadedFileAction, and the @FileItemUpload annotation).

Package Name Changes

All classes from the org.apache.commons.fileupload package have been moved. If your custom actions process uploaded files, you must update the package imports:

// BEFORE
import org.apache.commons.fileupload.FileItem;

// AFTER
import org.apache.commons.fileupload2.core.FileItem;

DiskFileItemFactory and JakartaServletFileUpload

If your application instantiates factories or upload handlers manually, note that DiskFileItemFactory now uses a builder pattern, and ServletFileUpload has been replaced with the Jakarta-specific implementation:

// BEFORE
DiskFileItemFactory factory = new DiskFileItemFactory();
ServletFileUpload upload = new ServletFileUpload(factory);

// AFTER
DiskFileItemFactory factory = DiskFileItemFactory.builder().get();
JakartaServletFileUpload upload = new JakartaServletFileUpload(factory);

DWR removed

Direct Web Remoting (DWR) has been completely removed from OpenXava. DWR was a library unmaintained since 2016, which represented a significant security risk. Furthermore, DWR does not support the Jakarta namespace, meaning it cannot function on modern servlet containers like Tomcat 11. Therefore, DWR is neither included nor can it be added to OpenXava 8.0 projects since it will not work.

DWR was primarily used internally by OpenXava, so standard projects should not be affected. However, if you have developed your own custom editor by copying an OpenXava editor that relied on DWR, you must migrate it. A simple and robust option is to migrate from DWR to JavaScript fetch combined with a standard Java Servlet.

You can refer to the following example of this migration in OpenXava:

  • Before (with DWR): Tree.java and treeEditor.js
  • After (with Fetch + Servlet): TreeServlet.java and treeEditor.js

Using an AI assistant to migrate your DWR logic to modern fetch + servlet is highly recommended and usually yields excellent results.
