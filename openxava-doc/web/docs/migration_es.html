openxava / documentación / Migración

¿Cómo actualizar tu proyecto a la última versión de OpenXava?

Edita el archivo pom.xml de tu proyecto y cambia el valor de la propiedad openxava.version para que apunte a la última versión de OpenXava, así:

<properties>
    <openxava.version>8.0</openxava.version>
    ...
</properties>

Reconstruye tu proyecto con: mvn clean package

Si tu versión actual es anterior a la 8.0 deberías seguir las instrucciones de la sección Para migrar de OpenXava 7.7.3 a OpenXava 8.0 primero.

Para finalizar, revisa las instrucciones de abajo para adaptar tu código.

Para migrar de OpenXava 7.7.3 a OpenXava 8.0

Java 17 como versión mínima

Ahora Java 17 es la versión mínima soportada. Si tu proyecto todavía usa Java 8 u 11 deberías actualizarlo para que compile con Java 17, 21 o 25. Para ello modifica el pom.xml:

<properties>
    <maven.compiler.source>17</maven.compiler.source>
    <maven.compiler.target>17</maven.compiler.target>
    ...
</properties>

Asegúrate de tener instalado un JDK 17 (o superior) en tu sistema y configurado en tu IDE.

Puedes consultar la documentación oficial de Java de migración a Java 17 y si usas Java 8 la documentación de migración desde Java 8.

Eliminar la dependencia openxava-7.7-chat-jdk17

Si estabas usando la funcionalidad de chat con IA en la versión 7.7, tenías que añadir la dependencia openxava-7.7-chat-jdk17 a tu pom.xml. En la versión 8.0 esta dependencia ya no es necesaria porque la funcionalidad de chat ahora está incluida en el núcleo de OpenXava.

Elimina la siguiente dependencia de tu pom.xml:

<!-- Elimina esta dependencia -->
<dependency>
    <groupId>org.openxava</groupId>
    <artifactId>openxava-7.7-chat-jdk17</artifactId>
    <version>${openxava.version}</version>
</dependency>

Si no la eliminas, tu build fallará porque este artefacto no existe en la versión 8.0.

Del lanzador AppServer a Spring Boot

En OpenXava 7.7 la aplicación se arrancaba con una clase lanzadora personalizada ubicada en el paquete run, que llamaba a AppServer.run() para iniciar un Tomcat embebido, y se ejecutaba con mvn exec:java (configurado mediante el exec-maven-plugin). En OpenXava 8.0 la aplicación es una aplicación Spring Boot estándar que se arranca con mvn spring-boot:run.

Sustituir la clase lanzadora

Elimina tu antigua clase lanzadora (la del paquete run) y crea una clase de aplicación Spring Boot que extienda org.openxava.spring.OpenXavaApplication. Colócala en el paquete raíz de tu aplicación:

// ANTES: src/main/java/com/yourcompany/yourapp/run/yourapp.java
package com.yourcompany.yourapp.run;

import org.openxava.util.*;

public class yourapp {
    public static void main(String[] args) throws Exception {
        DBServer.start("yourapp-db");
        AppServer.run("yourapp");
    }
}

// DESPUÉS: src/main/java/com/yourcompany/yourapp/YourappApplication.java
package com.yourcompany.yourapp;

import org.openxava.util.DBServer;
import org.openxava.spring.OpenXavaApplication;
import org.springframework.boot.SpringApplication;
import org.springframework.boot.autoconfigure.SpringBootApplication;

@SpringBootApplication
public class YourappApplication extends OpenXavaApplication {
    public static void main(String[] args) throws Exception {
        DBServer.start("yourapp-db"); // Solo para el HSQLDB incluido, elimínalo si usas tu propia base de datos
        SpringApplication.run(YourappApplication.class, args);
    }
}

Extender OpenXavaApplication (que a su vez extiende SpringBootServletInitializer) hace que la aplicación funcione tanto al lanzarla con Spring Boot como al desplegarla como WAR en un contenedor de servlets externo.

Sustituir exec-maven-plugin por spring-boot-maven-plugin

En tu pom.xml, elimina el exec-maven-plugin y añade el spring-boot-maven-plugin apuntando a tu nueva clase de aplicación:

<!-- ELIMINA este plugin -->
<plugin>
    <groupId>org.codehaus.mojo</groupId>
    <artifactId>exec-maven-plugin</artifactId>
    <version>3.1.0</version>
    <configuration>
        <mainClass>${project.groupId}.${project.artifactId}.run.${project.artifactId}</mainClass>
    </configuration>
</plugin>

<!-- AÑADE este plugin -->
<plugin>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-maven-plugin</artifactId>
    <version>4.1.0</version>
    <executions>
        <execution>
            <goals>
                <goal>repackage</goal>
            </goals>
        </execution>
    </executions>
    <configuration>
        <mainClass>com.yourcompany.yourapp.YourappApplication</mainClass>
    </configuration>
</plugin>

Además, como ahora el servidor embebido lo proporciona Spring Boot, no debes excluir los jars de Tomcat de tu WAR. Elimina la línea WEB-INF/lib/tomcat-*.jar de los packagingExcludes del maven-war-plugin si está presente.

Mover la configuración de la base de datos a application.properties

En OpenXava 7.7 el datasource se declaraba mediante JNDI en src/main/webapp/META-INF/context.xml. Con Spring Boot eliminas ese fichero y defines el datasource en src/main/resources/application.properties en su lugar. OpenXava enlaza este datasource de Spring con el nombre JNDI usado en persistence.xml, así que no necesitas cambiar persistence.xml.

server.servlet.context-path=/yourapp
server.tomcat.additional-tld-skip-patterns=htmlunit-*.jar,jetty-*.jar

spring.datasource.username=sa
spring.datasource.password=
spring.datasource.hikari.maximum-pool-size=20
spring.datasource.hikari.minimum-idle=5
spring.datasource.hikari.connection-timeout=10000

# HSQLDB
spring.datasource.url=jdbc:hsqldb:hsql://localhost:1666
spring.datasource.driver-class-name=org.hsqldb.jdbcDriver

Varios datasources

Si tu aplicación usa más de un datasource (por ejemplo porque tiene persistence-unit secundarias en persistence.xml), declara los adicionales en el mismo application.properties usando el prefijo openxava.datasources.<nombre>.*. El datasource por defecto sigue usando spring.datasource.*. OpenXava crea cada datasource adicional y lo enlaza al nombre JNDI declarado en su propiedad jndi, que debe coincidir con el usado en persistence.xml. No hace falta ninguna clase de configuración Java:

openxava.datasources.companya.jndi=jdbc/companyaDS
openxava.datasources.companya.url=jdbc:hsqldb:file:data/company-a-db
openxava.datasources.companya.username=sa
openxava.datasources.companya.password=
openxava.datasources.companya.driver-class-name=org.hsqldb.jdbcDriver
openxava.datasources.companya.maximum-pool-size=20
openxava.datasources.companya.minimum-idle=5
openxava.datasources.companya.connection-timeout=10000

openxava.datasources.companyb.jndi=jdbc/companybDS
openxava.datasources.companyb.url=jdbc:hsqldb:file:data/company-b-db
openxava.datasources.companyb.username=sa
openxava.datasources.companyb.password=
openxava.datasources.companyb.driver-class-name=org.hsqldb.jdbcDriver

Así toda la configuración de base de datos de tu aplicación queda en un único fichero declarativo, equivalente al antiguo context.xml pero al estilo estándar de Spring Boot.

Ejecutar la aplicación

Para arrancar la aplicación usa:

mvn spring-boot:run

El antiguo comando mvn exec:java ya no se aplica.

Servlets con anotación @WebServlet en lugar de web.xml

Cuando la aplicación se ejecuta con el Tomcat embebido de Spring Boot, el archivo src/main/webapp/WEB-INF/web.xml no se procesa. Esto significa que cualquier servlet, filtro o listener declarado en web.xml no se registrará. Debes declarar tus servlets personalizados usando la anotación @WebServlet (y @WebFilter, @WebListener para filtros y listeners) en la propia clase Java:

<!-- ANTES: src/main/webapp/WEB-INF/web.xml -->
<servlet>
    <servlet-name>testServlet</servlet-name>
    <servlet-class>com.yourcompany.yourapp.servlets.TestServlet</servlet-class>
</servlet>
<servlet-mapping>
    <servlet-name>testServlet</servlet-name>
    <url-pattern>/test</url-pattern>
</servlet-mapping>
// DESPUÉS: src/main/java/com/yourcompany/yourapp/servlets/TestServlet.java
import jakarta.servlet.annotation.WebServlet;
import jakarta.servlet.http.HttpServlet;

@WebServlet("/test")
public class TestServlet extends HttpServlet {
    // ...
}

La auto-configuración de OpenXava ya escanea los paquetes org.openxava y com.openxava en busca de anotaciones @WebServlet, @WebFilter y @WebListener, por lo que si tus servlets están en esos paquetes se detectarán automáticamente. Si tus servlets están en un paquete diferente, añade @ServletComponentScan a tu clase de aplicación con los paquetes base correspondientes.

Dado que web.xml ya no es necesario, deberías eliminar el archivo src/main/webapp/WEB-INF/web.xml. Cuando la aplicación se despliega como WAR en un contenedor de servlets externo, las anotaciones son procesadas por el contenedor automáticamente, por lo que web.xml no es necesario en ningún escenario.

Jakarta Persistence 3.2 con Hibernate 7.2

OpenXava 8.0 usa Jakarta Persistence 3.2 y Hibernate 7.2. Esto implica cambios significativos respecto a las versiones anteriores. A continuación detallamos los problemas más comunes que puedes encontrar al migrar tu aplicación.

Puedes consultar la guía de migración oficial de Hibernate para más información.

Cambio de javax.persistence a jakarta.persistence

El cambio principal es el reemplazo del namespace javax por jakarta en todas las anotaciones de JPA. Esto afecta a tus entidades, clases clave, y cualquier código que use anotaciones de persistencia.

// ANTES
import javax.persistence.Entity;
import javax.persistence.Id;
import javax.persistence.ManyToOne;

// DESPUÉS
import jakarta.persistence.Entity;
import jakarta.persistence.Id;
import jakarta.persistence.ManyToOne;

Además, si usas import org.hibernate.annotations.*; en tus archivos, esto puede causar ambigüedad entre org.openxava.annotations.View y org.hibernate.annotations.View. La solución es usar imports específicos:

// ANTES
import org.hibernate.annotations.*;

// DESPUÉS
import org.hibernate.annotations.Columns;
import org.hibernate.annotations.GenericGenerator;
import org.hibernate.annotations.Parameter;
import org.hibernate.annotations.Type;

persistence.xml

El archivo persistence.xml requiere actualizar el namespace a la versión 3.1 de Jakarta Persistence:

<persistence version="3.1" 
    xmlns="https://jakarta.ee/xml/ns/persistence"
    xmlns:xsi="https://www.w3.org/2001/XMLSchema-instance"
    xsi:schemaLocation="https://jakarta.ee/xml/ns/persistence
    https://jakarta.ee/xml/ns/persistence/persistence_3_1.xsd">

Formato del nombre JNDI de la fuente de datos

El formato del nombre JNDI de la fuente de datos en persistence.xml ha cambiado. En OpenXava 7.x el nombre usaba java:// como prefijo, pero en OpenXava 8.0 debes usar java: (sin la doble barra). El formato anterior con // no funciona en 8.0:

<!-- ANTES (7.x) -->
<non-jta-data-source>java://comp/env/jdbc/yourappDS</non-jta-data-source>

<!-- DESPUÉS (8.0) -->
<non-jta-data-source>java:comp/env/jdbc/yourappDS</non-jta-data-source>

NonBatchingBatcherFactory eliminado

La propiedad hibernate.jdbc.factory_class con valor org.hibernate.jdbc.NonBatchingBatcherFactory ya no es válida en Hibernate 7. Elimina esta propiedad de tu persistence.xml.

@Column(scale/precision) y @Digits en tipos float/double

Hibernate 7 ya no permite usar @Column(scale=..., precision=...) ni @Digits(integer=..., fraction=...) en tipos float o double:

// ERROR en Hibernate 7
@Column(scale=7, precision=7)
private double axisY;

Solución: Usa @DecimalMax de Jakarta Validator en su lugar:

// Solución válida
@DecimalMax("0.9999999")
private double axisY;

Uniones anidadas (Nested Joins)

En Hibernate 6+ las uniones anidadas deben usar la ruta completa desde la entidad raíz. No puedes usar el alias de un join previo como prefijo para un join anidado.

// INCORRECTO (Falla en Hibernate 6+)
SELECT e FROM Product2 e
LEFT JOIN e.subfamily e_subfamily
LEFT JOIN e_subfamily.family e_subfamily_family -- ¡Error!

// CORRECTO (Ruta completa desde la entidad raíz)
SELECT e FROM Product2 e
LEFT JOIN e.subfamily e_subfamily
LEFT JOIN e.subfamily.family e_subfamily_family -- ¡Correcto!

Atributos no cualificados en consultas JPQL

Si ordenas o filtras por un campo común (como name, id, description, code, etc.) y la consulta tiene JOINs, Hibernate lanzará la excepción Ambiguous unqualified attribute reference. Debes calificar siempre las referencias a propiedades con el alias de su entidad:

// INCORRECTO (Falla en Hibernate 6+)
SELECT e FROM Traveler e
LEFT JOIN e.lastJourney e_lastJourney
ORDER BY name -- ¡Error! 'name' existe en Traveler y en Journey

// CORRECTO
SELECT e FROM Traveler e
LEFT JOIN e.lastJourney e_lastJourney
ORDER BY e.name -- ¡Correcto!

Claves compuestas (@IdClass) con relaciones

Con Hibernate 6/7, las clases @IdClass que contienen campos que son relaciones @ManyToOne deben adaptarse:

  1. Cambia el tipo del campo de la relación por su clave primaria (ej. String en lugar de State)
  2. Elimina las anotaciones de mapeo relacional (@ManyToOne, @JoinColumn, etc.) de la clase clave
// ANTES (No compatible con Hibernate 6/7)
public class CityKey implements Serializable {
    @ManyToOne(fetch=FetchType.LAZY)
    @JoinColumn(name="STATE", referencedColumnName="ID")
    private State state;
    private int code;
}

// DESPUÉS (Compatible)
public class CityKey implements Serializable {
    private String state; // Tipo de la clave primaria de State
    private int code;
}

La entidad principal no requiere cambios en sus anotaciones @Id o @ManyToOne.

Expresiones booleanas en JPQL

En Hibernate 6+ los literales booleanos se compilan directamente a literales booleanos del dialecto SQL, lo que puede romper la compatibilidad con bases de datos que usan tipos CHAR para booleanos (como OpenXava con SiNoType).

// INCORRECTO (Genera error en Hibernate 6+)
Query query = XPersistence.getManager()
    .createQuery("from Invoice as o where o.paid = true");

// CORRECTO (Usa parámetros)
Query query = XPersistence.getManager()
    .createQuery("from Invoice as o where o.paid = :paid");
query.setParameter("paid", true);

Tipos personalizados (UserType/CompositeUserType)

Los tipos personalizados requieren varios cambios en Hibernate 7:

  • PrimitiveType/DiscriminatorType: Elimina estas interfaces de la declaración de tus clases
  • @TypeDef: Eliminado, usa @Convert o @CompositeType
  • @Type(type="org.hibernate.type.YesNoType"): Cambia a @Convert(converter = org.hibernate.type.YesNoConverter.class)
  • @Type(type="org.openxava.types.SiNoType"): Cambia a @Type(org.openxava.types.SiNoType.class)

Los tipos personalizados (UserType y CompositeUserType) prácticamente requieren reescribirse para la nueva API de Hibernate 7. Los cambios son profundos e incluyen la eliminación de métodos, cambios en las firmas, y nuevo uso de @Embeddable. Consulta la documentación de Hibernate para los detalles completos.

Para CompositeUserType con múltiples columnas, cambia @Columns por @AttributeOverrides:

// ANTES
@CompositeType(org.openxava.types.Date3Type.class)
@Columns(columns = {
    @Column(name="YEARDELIVERY"),
    @Column(name="MONTHDELIVERY"),
    @Column(name="DAYDELIVERY")
})

// DESPUÉS
@CompositeType(org.openxava.types.Date3Type.class)
@AttributeOverrides({
    @AttributeOverride(name="year", column=@Column(name="YEARDELIVERY")),
    @AttributeOverride(name="month", column=@Column(name="MONTHDELIVERY")),
    @AttributeOverride(name="day", column=@Column(name="DAYDELIVERY"))
})

Character a String en consultas nativas

Las consultas nativas a columnas CHAR ahora devuelven Character en lugar de String. Usa toString() para convertir:

// Antes (Hibernate 5.6)
String value = (String) query.getSingleResult();

// Después (Hibernate 7.2)
String value = query.getSingleResult().toString();

Cambio en la longitud del UUID

En Hibernate 5 el UUID por defecto era de 32 caracteres (sin guiones), pero en Hibernate 7 es de 36 caracteres (con guiones). Si tienes tablas existentes con columnas dimensionadas para 32 caracteres, necesitas usar la anotación @UUID32 de OpenXava para mantener el formato de 32 caracteres:

// ANTES (Hibernate 5)
@Id @GeneratedValue(generator="system-uuid") @Hidden 
@GenericGenerator(name="system-uuid", strategy = "uuid")
@Column(length=32)
private String id;

// DESPUÉS (Hibernate 7 con @UUID32)
@Id @Hidden 
@UUID32
@Column(length=32)
private String id;

Import: import org.openxava.annotations.UUID32;

Jakarta Validator 3.1 con Hibernate Validator 9.1

OpenXava 8.0 usa Jakarta Validator 3.1 y Hibernate Validator 9.1. Esto implica cambios en las anotaciones de validación.

Puedes consultar la guía de migración oficial de Hibernate Validator para más información.

Cambio de javax.validation a jakarta.validation

Cambia los imports de validación:

// ANTES
import javax.validation.constraints.*;
import javax.validation.Payload;

// DESPUÉS
import jakarta.validation.constraints.*;
import jakarta.validation.Payload;

Jakarta Servlet 6.1 con Tomcat 11

OpenXava 8.0 ha migrado toda su capa web a la moderna especificación Jakarta Servlet 6.1 sobre Tomcat 11. Esto requiere ajustar cualquier componente servlet personalizado, vistas JSP y configuraciones web en tu aplicación.

Cambio de javax.servlet a jakarta.servlet

Las clases estándar de Java Servlet han cambiado su espacio de nombres de paquetes. Si tu aplicación define filtros, servlets, listeners u otras clases personalizadas relacionadas con servlets, debes renombrar los imports:

// ANTES
import javax.servlet.*;
import javax.servlet.http.*;
import javax.servlet.jsp.*;

// DESPUÉS
import jakarta.servlet.*;
import jakarta.servlet.http.*;
import jakarta.servlet.jsp.*;

Eliminación de web.xml

Dado que OpenXava 8.0 usa Spring Boot, el archivo web.xml ya no se procesa cuando se ejecuta con el Tomcat embebido. Como se explica en la sección de migración a Spring Boot, deberías eliminar src/main/webapp/WEB-INF/web.xml y declarar tus servlets, filtros y listeners usando anotaciones (@WebServlet, @WebFilter, @WebListener) en su lugar.

Si por alguna razón necesitas mantener web.xml (por ejemplo para un despliegue solo como WAR en un contenedor de servlets externo), debe actualizarse para hacer referencia al espacio de nombres de Servlet 6.1:

<web-app xmlns="https://jakarta.ee/xml/ns/jakartaee"
         xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
         xsi:schemaLocation="https://jakarta.ee/xml/ns/jakartaee 
         https://jakarta.ee/xml/ns/jakartaee/web-app_6_1.xsd"
         version="6.1">

JSTL y taglibs de JSP

Las URIs de la biblioteca de etiquetas estándar de JavaServer Pages (JSTL) han cambiado de sus ubicaciones legadas en sun.com al nuevo espacio de nombres jakarta.tags. Si tienes archivos JSP personalizados, debes actualizar las directivas taglib:

<%-- ANTES --%>
<%@ taglib uri="http://java.sun.com/jsp/jstl/core" prefix="c" %>
<%@ taglib uri="http://java.sun.com/jsp/jstl/fmt" prefix="fmt" %>
<%@ taglib uri="http://java.sun.com/jsp/jstl/functions" prefix="fn" %>

<%-- DESPUÉS --%>
<%@ taglib uri="jakarta.tags.core" prefix="c" %>
<%@ taglib uri="jakarta.tags.fmt" prefix="fmt" %>
<%@ taglib uri="jakarta.tags.functions" prefix="fn" %>

Commons FileUpload 2

OpenXava 8.0 ha actualizado la biblioteca de subida de archivos a Apache Commons FileUpload 2. Esta versión incluye cambios importantes en los nombres de paquetes y el diseño de la API, particularmente alrededor de la interfaz FileItem, que se expone en varias APIs públicas de OpenXava (como WithFileItem, IProcessLoadedFileAction y la anotación @FileItemUpload).

Cambios en los nombres de paquetes

Todas las clases del paquete org.apache.commons.fileupload se han movido. Si tus acciones personalizadas procesan archivos subidos, debes actualizar los imports:

// ANTES
import org.apache.commons.fileupload.FileItem;

// DESPUÉS
import org.apache.commons.fileupload2.core.FileItem;

DiskFileItemFactory y JakartaServletFileUpload

Si tu aplicación instancia fábricas o controladores de subida manualmente, ten en cuenta que DiskFileItemFactory ahora utiliza un patrón builder, y ServletFileUpload ha sido reemplazado por la implementación específica para Jakarta:

// ANTES
DiskFileItemFactory factory = new DiskFileItemFactory();
ServletFileUpload upload = new ServletFileUpload(factory);

// DESPUÉS
DiskFileItemFactory factory = DiskFileItemFactory.builder().get();
JakartaServletFileUpload upload = new JakartaServletFileUpload(factory);

JAX-RS 3.1 y Jakarta JSON Binding 3.0

OpenXava 8.0 ha migrado el cliente REST (JAX-RS, implementado por Jersey) y la API de JSON Binding al espacio de nombres Jakarta. JAX-RS se ha actualizado a 3.1 (Jersey 3.1) y Jakarta JSON Binding a 3.0. Si tu aplicación usa el cliente JAX-RS o las anotaciones de JSON-B (por ejemplo en un servicio que consume un servicio web REST), el único cambio necesario es actualizar los imports del espacio de nombres javax a jakarta:

// ANTES
import javax.ws.rs.client.*;
import javax.json.bind.annotation.*;

// DESPUÉS
import jakarta.ws.rs.client.*;
import jakarta.json.bind.annotation.*;

La API de estas clases (como ClientBuilder, @JsonbProperty, etc.) no cambia, por lo que no es necesario realizar ningún otro cambio en tu código.

DWR eliminado

Direct Web Remoting (DWR) se ha eliminado por completo de OpenXava. DWR era una librería sin mantenimiento desde 2016, lo que representaba un problema de seguridad importante. Además, DWR no soporta el espacio de nombres de Jakarta, por lo que no podría funcionar en contenedores de servlets modernos como Tomcat 11. Esto significa que DWR no está incluido, pero tampoco se puede añadir a proyectos de OpenXava 8 porque no funcionará.

DWR se utilizaba principalmente de forma interna en OpenXava, por lo que los proyectos estándar no deberían verse afectados. Sin embargo, si has desarrollado tu propio editor personalizado copiando un editor de OpenXava que dependía de DWR, deberás migrarlo. Una opción sencilla y robusta es migrar de DWR a fetch de JavaScript con un Servlet Java simple y llano.

Puedes consultar el siguiente ejemplo de esta migración en OpenXava:

  • Antes (con DWR): Tree.java y treeEditor.js
  • Después (con Fetch + Servlet): TreeServlet.java y treeEditor.js

Pedir a un asistente de IA que migre tu lógica de DWR a fetch + servlet es altamente recomendable y suele dar muy buen resultado.

Para migrar a OpenXava 7.7.3

Para migrar a OpenXava 7.7.3 desde cualquier versión anterior de OpenXava:

Sigue las instrucciones de migración para versiones anteriores
