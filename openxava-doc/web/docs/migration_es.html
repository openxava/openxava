¿Cómo actualizar tu proyecto a la última versión de OpenXava?

Edita el archivo pom.xml de tu proyecto y cambia el valor de la propiedad openxava.version para que apunte a la última versión de OpenXava, así:

<properties>
    <openxava.version>8.0</openxava.version>
    ...
</properties>

Reconstruye tu proyecto con: mvn clean package

Si tu versión actual es anterior a la 8.0 deberías seguir las instrucciones de la sección Para migrar de OpenXava 7.7.3 a OpenXava 8.0 primero.

Para finalizar, revisa las instrucciones de abajo para adaptar tu código.

Para migrar de OpenXava 7.7.3 a OpenXava 8.0

Java 17 como versión mínima

Ahora Java 17 es la versión mínima soportada. Si tu proyecto todavía usa Java 8 u 11 deberías actualizarlo para que compile con Java 17, 21 o 25. Para ello modifica el pom.xml:

<properties>
    <maven.compiler.source>17</maven.compiler.source>
    <maven.compiler.target>17</maven.compiler.target>
    ...
</properties>

Asegúrate de tener instalado un JDK 17 (o superior) en tu sistema y configurado en tu IDE.

Puedes consultar la documentación oficial de Java de migración a Java 17 y si usas Java 8 la documentación de migración desde Java 8.

Jakarta Persistence 3.2 con Hibernate 7.2

OpenXava 8.0 usa Jakarta Persistence 3.2 y Hibernate 7.2. Esto implica cambios significativos respecto a las versiones anteriores. A continuación detallamos los problemas más comunes que puedes encontrar al migrar tu aplicación.

Puedes consultar la guía de migración oficial de Hibernate para más información.

Cambio de javax.persistence a jakarta.persistence

El cambio principal es el reemplazo del namespace javax por jakarta en todas las anotaciones de JPA. Esto afecta a tus entidades, clases clave, y cualquier código que use anotaciones de persistencia.

// ANTES
import javax.persistence.Entity;
import javax.persistence.Id;
import javax.persistence.ManyToOne;

// DESPUÉS
import jakarta.persistence.Entity;
import jakarta.persistence.Id;
import jakarta.persistence.ManyToOne;

Además, si usas import org.hibernate.annotations.*; en tus archivos, esto puede causar ambigüedad entre org.openxava.annotations.View y org.hibernate.annotations.View. La solución es usar imports específicos:

// ANTES
import org.hibernate.annotations.*;

// DESPUÉS
import org.hibernate.annotations.Columns;
import org.hibernate.annotations.GenericGenerator;
import org.hibernate.annotations.Parameter;
import org.hibernate.annotations.Type;

persistence.xml

El archivo persistence.xml requiere actualizar el namespace a la versión 3.1 de Jakarta Persistence:

<persistence version="3.1" 
    xmlns="https://jakarta.ee/xml/ns/persistence"
    xmlns:xsi="https://www.w3.org/2001/XMLSchema-instance"
    xsi:schemaLocation="https://jakarta.ee/xml/ns/persistence
    https://jakarta.ee/xml/ns/persistence/persistence_3_1.xsd">

NonBatchingBatcherFactory eliminado

La propiedad hibernate.jdbc.factory_class con valor org.hibernate.jdbc.NonBatchingBatcherFactory ya no es válida en Hibernate 7. Elimina esta propiedad de tu persistence.xml.

@Column(scale/precision) y @Digits en tipos float/double

Hibernate 7 ya no permite usar @Column(scale=..., precision=...) ni @Digits(integer=..., fraction=...) en tipos float o double:

// ERROR en Hibernate 7
@Column(scale=7, precision=7)
private double axisY;

Solución: Usa @DecimalMax de Jakarta Validator en su lugar:

// Solución válida
@DecimalMax("0.9999999")
private double axisY;

Uniones anidadas (Nested Joins)

En Hibernate 6+ las uniones anidadas deben usar la ruta completa desde la entidad raíz. No puedes usar el alias de un join previo como prefijo para un join anidado.

// INCORRECTO (Falla en Hibernate 6+)
SELECT e FROM Product2 e
LEFT JOIN e.subfamily e_subfamily
LEFT JOIN e_subfamily.family e_subfamily_family -- ¡Error!

// CORRECTO (Ruta completa desde la entidad raíz)
SELECT e FROM Product2 e
LEFT JOIN e.subfamily e_subfamily
LEFT JOIN e.subfamily.family e_subfamily_family -- ¡Correcto!

Atributos no cualificados en consultas JPQL

Si ordenas o filtras por un campo común (como name, id, description, code, etc.) y la consulta tiene JOINs, Hibernate lanzará la excepción Ambiguous unqualified attribute reference. Debes calificar siempre las referencias a propiedades con el alias de su entidad:

// INCORRECTO (Falla en Hibernate 6+)
SELECT e FROM Traveler e
LEFT JOIN e.lastJourney e_lastJourney
ORDER BY name -- ¡Error! 'name' existe en Traveler y en Journey

// CORRECTO
SELECT e FROM Traveler e
LEFT JOIN e.lastJourney e_lastJourney
ORDER BY e.name -- ¡Correcto!

Claves compuestas (@IdClass) con relaciones

Con Hibernate 6/7, las clases @IdClass que contienen campos que son relaciones @ManyToOne deben adaptarse:

  1. Cambia el tipo del campo de la relación por su clave primaria (ej. String en lugar de State)
  2. Elimina las anotaciones de mapeo relacional (@ManyToOne, @JoinColumn, etc.) de la clase clave
// ANTES (No compatible con Hibernate 6/7)
public class CityKey implements Serializable {
    @ManyToOne(fetch=FetchType.LAZY)
    @JoinColumn(name="STATE", referencedColumnName="ID")
    private State state;
    private int code;
}

// DESPUÉS (Compatible)
public class CityKey implements Serializable {
    private String state; // Tipo de la clave primaria de State
    private int code;
}

La entidad principal no requiere cambios en sus anotaciones @Id o @ManyToOne.

Expresiones booleanas en JPQL

En Hibernate 6+ los literales booleanos se compilan directamente a literales booleanos del dialecto SQL, lo que puede romper la compatibilidad con bases de datos que usan tipos CHAR para booleanos (como OpenXava con SiNoType).

// INCORRECTO (Genera error en Hibernate 6+)
Query query = XPersistence.getManager()
    .createQuery("from Invoice as o where o.paid = true");

// CORRECTO (Usa parámetros)
Query query = XPersistence.getManager()
    .createQuery("from Invoice as o where o.paid = :paid");
query.setParameter("paid", true);

Tipos personalizados (UserType/CompositeUserType)

Los tipos personalizados requieren varios cambios en Hibernate 7:

  • PrimitiveType/DiscriminatorType: Elimina estas interfaces de la declaración de tus clases
  • @TypeDef: Eliminado, usa @Convert o @CompositeType
  • @Type(type="org.hibernate.type.YesNoType"): Cambia a @Convert(converter = org.hibernate.type.YesNoConverter.class)
  • @Type(type="org.openxava.types.SiNoType"): Cambia a @Type(org.openxava.types.SiNoType.class)

Los tipos personalizados (UserType y CompositeUserType) prácticamente requieren reescribirse para la nueva API de Hibernate 7. Los cambios son profundos e incluyen la eliminación de métodos, cambios en las firmas, y nuevo uso de @Embeddable. Consulta la documentación de Hibernate para los detalles completos.

Para CompositeUserType con múltiples columnas, cambia @Columns por @AttributeOverrides:

// ANTES
@CompositeType(org.openxava.types.Date3Type.class)
@Columns(columns = {
    @Column(name="YEARDELIVERY"),
    @Column(name="MONTHDELIVERY"),
    @Column(name="DAYDELIVERY")
})

// DESPUÉS
@CompositeType(org.openxava.types.Date3Type.class)
@AttributeOverrides({
    @AttributeOverride(name="year", column=@Column(name="YEARDELIVERY")),
    @AttributeOverride(name="month", column=@Column(name="MONTHDELIVERY")),
    @AttributeOverride(name="day", column=@Column(name="DAYDELIVERY"))
})

Character a String en consultas nativas

Las consultas nativas a columnas CHAR ahora devuelven Character en lugar de String. Usa toString() para convertir:

// Antes (Hibernate 5.6)
String value = (String) query.getSingleResult();

// Después (Hibernate 7.2)
String value = query.getSingleResult().toString();

Jakarta Validator 3.1 con Hibernate Validator 9.1

OpenXava 8.0 usa Jakarta Validator 3.1 y Hibernate Validator 9.1. Esto implica cambios en las anotaciones de validación.

Puedes consultar la guía de migración oficial de Hibernate Validator para más información.

Cambio de javax.validation a jakarta.validation

Cambia los imports de validación:

// ANTES
import javax.validation.constraints.*;
import javax.validation.Payload;

// DESPUÉS
import jakarta.validation.constraints.*;
import jakarta.validation.Payload;

DWR eliminado

Direct Web Remoting (DWR) se ha eliminado por completo de OpenXava. DWR era una librería sin mantenimiento desde 2016, lo que representaba un problema de seguridad importante. Además, DWR no soporta el espacio de nombres de Jakarta, por lo que no podría funcionar en contenedores de servlets modernos como Tomcat 11. Esto significa que DWR no está incluido, pero tampoco se puede añadir a proyectos de OpenXava 8 porque no funcionará.

DWR se utilizaba principalmente de forma interna en OpenXava, por lo que los proyectos estándar no deberían verse afectados. Sin embargo, si has desarrollado tu propio editor personalizado copiando un editor de OpenXava que dependía de DWR, deberás migrarlo. Una opción sencilla y robusta es migrar de DWR a fetch de JavaScript con un Servlet Java simple y llano.

Puedes consultar el siguiente ejemplo de esta migración en OpenXava:

  • Antes (con DWR): Tree.java y treeEditor.js
  • Después (con Fetch + Servlet): TreeServlet.java y treeEditor.js

Pedir a un asistente de IA que migre tu lógica de DWR a fetch + servlet es altamente recomendable y suele dar muy buen resultado.
